# Maestro Live Verification — 2026-04-26

## Result: Live API tested via kubectl port-forward; bugs found and fixed

Live testing was performed against the GKE cluster at 34.175.77.98, Maestro
namespace, using kubectl port-forward svc/maestro 18002:8000.

## Live API Findings

1. GET /api/maestro/v1/consumers → HTTP 200
   - Returns 1 consumer: cluster1 (id: 6f56f8da-d465-4c07-9291-58facdd78fde)

2. GET /api/maestro/v1/resources → HTTP 404
   - The /resources endpoint does NOT exist in this Maestro deployment.
   - Initial implementation was wrong: used /resources for List/Get/Delete.

3. GET /api/maestro/v1/resource-bundles → HTTP 200
   - Returns 326 bundles for consumer cluster1.
   - Resource name is in metadata.name, NOT a top-level "name" field.

4. GET /api/maestro/v1/resource-bundles?search=consumer_name%20%3D%20%27cluster1%27 → HTTP 200
   - SQL-like search filter works: consumer_name = 'cluster1'
   - 326 results for cluster1 consumer.

5. GET /api/maestro/v1/resource-bundles/<uuid> → HTTP 200
   - Direct lookup by UUID works.
   - Bundle name comes from metadata["name"], e.g. "019db4a7-91fa-721c-ba93-4d9af6592a67-cl-m-bad-api"

## Bugs Fixed

1. Resource struct: removed top-level "name" field, added Metadata map[string]string
   with name stored at metadata["name"] to match real API shape.

2. List(): changed from GET /resources?consumer_name=X
   to GET /resource-bundles?search=consumer_name%20%3D%20%27X%27

3. Get(): changed from GET /resources/<name>
   to GET /resource-bundles/<id>

4. Delete(): changed from DELETE /resources/<name>
   to DELETE /resource-bundles/<id>

5. tui command: replaced --api-server=<endpoint> (invalid flag)
   with MAESTRO_HTTP_ENDPOINT and MAESTRO_GRPC_ENDPOINT env vars
   (what maestro-cli actually reads per its root.go source).

## maestro-cli Flag Reference (github.com/openshift-hyperfleet/maestro-cli)

  Persistent flags: --http-endpoint, --grpc-endpoint, --grpc-insecure, --grpc-client-token
  Env vars: MAESTRO_HTTP_ENDPOINT, MAESTRO_GRPC_ENDPOINT, MAESTRO_GRPC_TOKEN
